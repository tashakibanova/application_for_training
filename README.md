# Прототип веб-формы заявки на обучение

Статус: прототип для эксперимента (см. `бриф_прототип_веб_формы_заявки.md`), собран и сверен по частям. Реальный деплой и сквозной прогон с настоящими Б24/DaData/Sheets ещё не выполнялись — см. «Что проверить перед запуском теста» ниже.

Схема данных между частями — единый источник правды: **[`CONTRACT.md`](./CONTRACT.md)**. Если что-то меняете в полях/эндпоинтах — правьте контракт первым.

## Состав

| Папка | Что это | Куда деплоится |
|---|---|---|
| `site/` | Статичная форма (HTML/CSS/JS, без сборки) | GitHub Pages / Netlify |
| `worker/` | Cloudflare Worker — прокси в Б24/Sheets, генерация xlsx | Cloudflare Workers |
| `scripts/` | Разовый Node-скрипт экспорта каталога курсов из Б24 | запускается локально вручную |
| `sheets/` | Google Apps Script — приёмник в Google Sheets | Google Apps Script Web App |

## Порядок деплоя (по брифу, раздел «Что нужно подготовить до старта разработки»)

1. **Google Sheets**: создать таблицу, задеплоить `sheets/Code.gs` как Web App — инструкция в `sheets/README.md`. Получите `SHEETS_WEBHOOK_URL` и придуманный `SHARED_SECRET`.
2. **Cloudflare Worker**: `cd worker`, `npm install`, `wrangler login`, затем:
   ```
   wrangler secret put BITRIX_WEBHOOK_URL
   wrangler secret put SHEETS_WEBHOOK_URL
   wrangler secret put SHEETS_WEBHOOK_SECRET
   ```
   Поправить `ALLOWED_ORIGIN` в `wrangler.toml` на реальный домен формы, `wrangler deploy`. Подробности — `worker/README.md`.
3. **Каталог курсов**: `BITRIX_WEBHOOK_URL=... node scripts/export-catalog.js` — сгенерирует `site/courses.json` (сейчас там лежит рабочий пример на 8 курсов для теста формы). Проверить, что `hours` подхватились — см. `scripts/README.md`.
4. **DaData**: получить бесплатный ключ на dadata.ru, вписать в `site/app.js` вместо `DADATA_TOKEN = 'REPLACE-ME-DADATA-TOKEN'`.
5. **Форма**: вписать реальный адрес Worker в `site/app.js` вместо `WORKER_URL = 'https://REPLACE-ME.workers.dev/submit'` (эндпоинт `/track` выводится из него автоматически). Задеплоить `site/` на GitHub Pages или Netlify.
6. Ссылки менеджерам — вручную добавлять `?deal=<ID сделки>` к адресу формы.

## Что проверить перед запуском теста (не проверялось агентами — нет доступа к реальным сервисам)

- **Сквозной прогон** `/submit` с реальным Bitrix24-вебхуком: приходит ли xlsx-вложение в комментарий сделки, не бьётся ли кириллица.
- **Поле часов в Б24**: `scripts/export-catalog.js` использует эвристику (`PROPERTY_HOURS` + запасные поля) — при первом запуске почти наверняка нужно будет поправить константу под реальную структуру каталога (подробно расписано в `scripts/README.md`).
- **SheetJS в Workers runtime**: код собран так, чтобы не зависеть от Node `Buffer` (`XLSX.write(..., {type:"base64"})` + `nodejs_compat` как страховка), но живьём не запускался — при первом деплое стоит сразу протестировать `/submit`.
- **DaData**: без ключа автоподбор по ИНН/адресу просто показывает пользователю понятное сообщение «заполните вручную» — форма не ломается, но проверить работу с реальным ключом всё равно нужно.

## Как решены открытые вопросы брифа (для прототипа, не окончательно)

1. **Заявка без `?deal=`** — уже покрыто основной логикой: пишется в лист «Незакреплённые заявки» с пометкой, разбирается менеджером вручную (как и написано в брифе). Дополнительно Worker подстраховывается: если `dealId` указан, но вызов Б24 упал по любой причине, заявка тоже падает в этот лист (а не теряется) — с пояснением ошибки в поле `note`. Более надёжный способ передачи ID сделки (короткая ссылка от Б24-виджета) — не делали, это отдельная фича вне прототипа.
2. **Незавершённое заполнение** — реализовано технически (минимально): `sendBeacon` на `beforeunload`/`pagehide`/скрытие вкладки шлёт `startedAt` и число тронутых строк слушателей в лист «Метрики эксперимента» со статусом `abandoned`. Best-effort — если браузер убьёт вкладку раньше события, строка просто не появится.
3. **Частота обновления `courses.json`** — оставлено как есть, ручной запуск скрипта; конкретную периодичность нужно определить вам по факту того, как часто меняется каталог во время теста (инженерного решения тут нет).

## Известные отклонения от точной буквы CONTRACT.md (обоснованные)

- Формат вложения файла в `crm.timeline.comment.add` — Worker использует `FILES: [[fileName, base64]]` (проверено по официальной документации Б24 REST), а не `fileContent`, как было в черновой формулировке контракта. Это правильный формат для реального API.
- В конверт `sheet: "unbound"` добавлено необязательное поле `xlsxBase64` (файл целиком в base64) сверх минимального набора из CONTRACT.md §4.1 — чтобы данные по неприкреплённым заявкам не терялись и файл можно было восстановить вручную.
