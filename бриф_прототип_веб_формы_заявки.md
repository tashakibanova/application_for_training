# Бриф: прототип веб-формы заявки на обучение

## Цель

Проверить, будут ли ЛПР (клиенты) вообще заполнять заявку на обучение в электронном виде вместо письма/Excel, и насколько это удобно. Это эксперимент, не production-функционал — приоритет на скорость запуска и простоту, а не на полноту автоматизации.

Из результата теста должно быть видно: сколько заявок начали заполнять, сколько дозаполнили до конца, сколько времени это заняло.

---

## Состав системы

Никакого своего бэкенда с базой данных не поднимаем. Схема: статичная страница + serverless-прокси + существующие системы (Б24, Google Sheets).

1. **Веб-форма** — статичная страница (HTML/CSS/JS, без сборки), хостинг GitHub Pages или Netlify.
2. **Каталог курсов** — `courses.json`, лежит рядом со страницей. Генерируется отдельным скриптом (Python/Node), который разово дёргает `catalog.product.list` через входящий вебхук Б24. Обновляется вручную по мере необходимости (для теста не нужен live-sync).
3. **DaData** (бесплатный тариф) — автоподбор наименования организации, адреса и КПП по ИНН, вызывается прямо из браузера.
4. **Cloudflare Worker** (serverless-функция) — принимает данные формы, генерирует xlsx, пишет комментарий с вложением в сделку Б24. Единственное место, где хранится токен вебхука Б24.
5. **Google Sheets** — запасной приёмник (если сделка не указана) и параллельный лог метрик самого эксперимента.

## Структура формы

### Блок 1 — данные организации (заполняется один раз)

- Полное наименование учреждения
- ИНН → автоподбор наименования/адреса/КПП через DaData suggest API
- Тип документа: Договор / Контракт / Муниципальный контракт
- 44-ФЗ или 223-ФЗ
- Нужен ИКЗ: да (с полем для номера) / нет
- Источник финансирования услуг
- Почтовый адрес для отправки документов (индекс, наименование учреждения, ФИО руководителя)
- ФИО руководителя
- Email и телефон для связи
- Способ получения оригиналов договора: СБИС / Контур
- Дополнительный комментарий (свободный текст)

### Блок 2 — слушатели (повторяющиеся строки, кнопка «добавить слушателя»)

- Курс — выпадающий список, значения из `courses.json`
- Кол-во часов — подставляется автоматически по выбранному курсу
- Дата проведения
- ФИО слушателя (как в паспорте)
- Email
- Личный телефон
- Должность — показывается только если выбранный курс относится к охране труда / пожарной безопасности / первой помощи
- Причина прохождения (для тех же курсов): первичная / очередная / внеочередная

Валидация обязательных полей на клиенте. Адаптивная вёрстка — форма часто заполняется с телефона.

## Логика отправки

Ссылка на форму, которую менеджер отправляет клиенту, содержит ID сделки Б24:
`форма.ru/?deal=12345`

При отправке формы:

1. Всегда генерировать **xlsx** (не CSV — из-за проблем с кодировкой кириллицы в CSV) со списком слушателей: одна строка — один слушатель, колонки соответствуют полям блока 2. Библиотека — SheetJS (работает в Cloudflare Workers).
2. POST с данными формы и сгенерированным xlsx уходит на Cloudflare Worker.
3. Worker вызывает `crm.timeline.comment.add` в сделку `dealId`:
   - файл xlsx прикрепляется в поле `FILES`;
   - текст комментария — краткая сводка (организация, число слушателей, курс(ы), дата отправки заявки).
4. Если `dealId` в query-параметре отсутствует — заявка (вместе со ссылкой на xlsx) пишется только в Google Sheet, с явной пометкой «не привязано к сделке», чтобы данные не терялись. Такие случаи разбираются вручную.
5. Независимо от исхода (привязано к сделке или нет) — в отдельный Google Sheet всегда дублируются метаданные заявки для анализа эксперимента: время начала заполнения, время отправки, число слушателей, факт незавершённого заполнения (если технически отслеживается, например через `beforeunload` или периодический автосейв черновика).

Токен вебхука Б24 и ключ DaData (если требует ограничения по домену) хранятся только в переменных окружения Cloudflare Worker — никогда не в коде фронтенда.

## Что нужно подготовить до старта разработки

1. Входящий вебхук Б24 с правами на `catalog` (чтение каталога) и `crm` (запись комментариев и файлов в сделки).
2. Скрипт экспорта каталога курсов в `courses.json`.
3. Бесплатный API-ключ DaData.
4. Google Sheet + Apps Script (или прямой доступ Cloudflare Worker к Google Sheets API) для двух листов: «Незакреплённые заявки» и «Метрики эксперимента».
5. Аккаунт Cloudflare (бесплатный тариф Workers достаточен для объёма теста).
6. Согласовать с менеджерами: ссылка на форму формируется вручную, добавлением `?deal=` из адресной строки открытой сделки Б24.

---

## Вне скоупа этого прототипа

- Обратная синхронизация каких-либо данных из Б24 в форму, кроме каталога курсов.
- Автоматическое создание контактов/зачисление слушателей на платформу — в этой версии данные только попадают в сделку, дальнейшую обработку менеджер делает вручную.
- Проверка дублей контактов.
- Живой (real-time) сикинг каталога — обновление курсов вручную запуском скрипта.

## Открытые вопросы

1. Что делать, если менеджер забыл указать `deal` в ссылке — устраивает ли ручная разборка таких случаев на время теста, или нужен более надёжный способ передачи ID сделки клиенту (например, короткая ссылка с редиректом, которую генерирует сам Б24-виджет)?
2. Нужно ли отслеживать незавершённое заполнение технически (автосейв черновика) или для целей эксперимента достаточно смотреть только факт «отправлено / не отправлено»?
3. Список курсов в `courses.json` — насколько часто он реально меняется на время теста, чтобы определить периодичность ручного обновления?
