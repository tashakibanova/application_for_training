# scripts/export-catalog.js

Разовый CLI-скрипт: тянет каталог курсов из Битрикс24 через входящий вебхук и
генерирует `site/courses.json` по схеме из `CONTRACT.md` §1.

Live-sync не нужен (см. бриф) — скрипт запускается вручную, когда каталог
курсов в Б24 меняется.

**Проверено на реальном каталоге eduregion.bitrix24.ru (2026-07-07): 798 курсов, выгрузка прошла успешно.**

## Источник данных

Курсы лежат в **iblockId 21** («Товарный каталог CRM») — это общий каталог на
весь Б24-аккаунт (там же, например, товары школы бизнеса и прочее непрофильное),
поэтому выборка идёт по конкретным разделам (`iblockSectionId`), а не по всему
iblock целиком. Разделы зашиты в константу `COURSE_SECTIONS` в начале
`export-catalog.js`:

| id раздела | Название | category |
|---|---|---|
| 719 | ЦДО КПК В продаже | `null` |
| 721 | ЦДО ПП В продаже | `null` |
| 1204 | ЕРЛ. Логопедия/дефектология | `null` |
| 1252 | ЕРЛ. Охрана труда | `labor_safety` |
| 1308 | ЕРЛ Сколково | `null` |
| 1330 | ЕРЛ Сколково: ПП | `null` |

Если состав актуальных разделов изменится (появится новый раздел с курсами,
старый архивируется и т.п.) — правьте `COURSE_SECTIONS` в начале файла.
Список всех разделов каталога: `catalog.section.list` с `filter.iblockId: 21`
(постранично, `start`).

## Как запускать

Нужен Node.js 18+ (используется встроенный `fetch`, зависимостей нет).

```bash
cd scripts
BITRIX_WEBHOOK_URL=https://your-portal.bitrix24.ru/rest/1/xxxxxxxxxxxxxxxx/ node export-catalog.js
```

По умолчанию результат пишется в `../site/courses.json`. Можно указать
другой путь первым аргументом:

```bash
BITRIX_WEBHOOK_URL=https://your-portal.bitrix24.ru/rest/1/xxxxxxxxxxxxxxxx/ node export-catalog.js /путь/до/courses.json
```

Вебхуку нужны права `catalog` (чтение каталога товаров) — см. CONTRACT.md §6.

На Windows PowerShell переменную окружения удобнее задать так:

```powershell
$env:BITRIX_WEBHOOK_URL = "https://your-portal.bitrix24.ru/rest/1/xxxxxxxxxxxxxxxx/"
node export-catalog.js
```

## Как часто перезапускать

Вручную, по мере изменения каталога курсов в Б24. Никакого расписания/крона
не настроено — это осознанное решение по брифу (для теста живой синк не нужен).
После запуска нужно закоммитить/задеплоить обновлённый `site/courses.json`
вместе со страницей (при пуше в GitHub Pages деплой формы обновится автоматически).

## Часы (`hours`)

Отдельного поля «часы» в карточке товара нет (проверено на реальном каталоге —
есть только безымянные `property1223..property1836`, в основном пустые). Часы
почти всегда зашиты прямо в название товара: `«... (108 ЧАСОВ)»`,
`«... (16 часов). СПЕЦИАЛЬНЫЙ»`. Скрипт вытаскивает их регуляркой
(`extractHours` в начале файла).

На реальном каталоге у **56 из 798** курсов часы в названии не нашлись
(например, курсы, измеряемые в неделях/месяцах, а не в часах, — это
нормально, не баг) — для них `hours: null`, форма в этом случае просто не
показывает подсказку по часам, поле остаётся пустым. Скрипт печатает список
таких ID при каждом запуске.

## Категории курсов

`category` определяется гибридно:

1. **По разделу** — основной сигнал (см. таблицу выше). Например, раздел
   «ЕРЛ. Охрана труда» всегда даёт `labor_safety`, даже если в названии
   товара нет слов «охрана труда» (на практике там частые формулировки вроде
   «Безопасные методы и приёмы выполнения газоопасных работ», без буквального
   упоминания «охрана труда»).
2. **По ключевым словам в названии** — запасной вариант (константа
   `CATEGORY_KEYWORDS`), ловит курсы по пожарной безопасности/первой помощи
   вне выделенных разделов, либо новые разделы, которые ещё не добавлены в
   `COURSE_SECTIONS`.

На реальном прогоне: `labor_safety` — 72, `fire_safety` — 3, `first_aid` — 5,
остальные 718 — `null` (курсы общего профиля).

## Пагинация

Скрипт учитывает постраничную выдачу `catalog.product.list`: пока в ответе
Б24 присутствует числовое поле `next`, запрашивается следующая страница
(`start = next`). Проверено на реальном каталоге — 798 товаров, 16 страниц по
50, пагинация отработала корректно.
