# scripts/export-catalog.js

Разовый CLI-скрипт: тянет каталог товаров (курсов) из Битрикс24 через входящий
вебхук и генерирует `site/courses.json` по схеме из `CONTRACT.md` §1.

Live-sync не нужен (см. бриф) — скрипт запускается вручную, когда каталог
курсов в Б24 меняется.

## Как запускать

Нужен Node.js 18+ (используется встроенный `fetch`, зависимостей нет).

```bash
cd scripts
BITRIX_WEBHOOK_URL=https://your-portal.bitrix24.ru/rest/1/xxxxxxxxxxxxxxxx/ node export-catalog.js
```

По умолчанию результат пишется в `../site/courses.json`. Можно указать
другой путь первым аргументом:

```bash
BITRIX_WEBHOOK_URL=https://your-portal.bitrix24.ru/rest/1/xxxxxxxxxxxxxxxx/ node export-catalog.js /путь/до/courses.json
```

Вебхуку нужны права `catalog` (чтение каталога товаров) — см. CONTRACT.md §6.

На Windows PowerShell переменную окружения удобнее задать так:

```powershell
$env:BITRIX_WEBHOOK_URL = "https://your-portal.bitrix24.ru/rest/1/xxxxxxxxxxxxxxxx/"
node export-catalog.js
```

## Как часто перезапускать

Вручную, по мере изменения каталога курсов в Б24. Никакого расписания/крона
не настроено — это осознанное решение по брифу (для теста живой синк не нужен).
После запуска нужно закоммитить/задеплоить обновлённый `site/courses.json`
вместе со страницей.

## Что проверить, если часы (`hours`) не подхватились

`catalog.product.list` в базовой конфигурации Б24 не возвращает "часы курса"
как отдельное стандартное поле — в разных порталах это либо пользовательское
свойство товара (property), либо часы вообще не заведены в каталоге и их
нужно будет вписывать в `courses.json` руками после генерации.

Скрипт пытается достать `hours` в таком порядке:

1. Поле `PROPERTY_HOURS` товара (константа `HOURS_PROPERTY_CODE` в начале
   `export-catalog.js`).
2. Список запасных полей `HOURS_FALLBACK_FIELDS` (там же).
3. Объект `PROPERTY_VALUES`/`PROPERTIES`, если Б24 вернул свойства товара
   в виде вложенной структуры — скрипт ищет там ключ, похожий на "hour"/"час".

Если после запуска в консоли скрипт пишет предупреждение
`"не удалось определить hours"` — значит, ни один из вариантов не сработал.
Что делать:

1. Открыть в Б24 карточку одного товара-курса (Каталог → Товары) и
   посмотреть, в каком именно свойстве хранится количество часов —
   вкладка "Настройка полей и разделов" в модуле каталога, либо
   через REST `catalog.product.get` (или `crm.product.get`, если каталог
   старого типа) с реальным ID товара — в ответе будет виден точный код
   свойства.
2. Поправить константу `HOURS_PROPERTY_CODE` (и при необходимости —
   массив `HOURS_FALLBACK_FIELDS`) в начале `export-catalog.js` на реальный
   код свойства.
3. Если часы в каталоге вообще не заведены как отдельное поле — самый
   быстрый путь для прототипа: сгенерировать `courses.json` без часов
   (`hours: null`) и дозаполнить значения руками в JSON-файле — это
   ожидаемый компромисс для разового прототипа, не требующий доработки
   скрипта.

## Категории курсов

`category` определяется по ключевым словам в названии товара — список
вынесен в константу `CATEGORY_KEYWORDS` в начале `export-catalog.js`, легко
редактируется. Если ни одно ключевое слово не совпало — `category: null`
(это ожидаемо для курсов общего профиля, не относящихся к охране труда/
пожарной безопасности/первой помощи).

## Пагинация

Скрипт учитывает постраничную выдачу `catalog.product.list`: пока в ответе
Б24 присутствует числовое поле `next`, запрашивается следующая страница
(`start = next`). Остановка — когда `next` в ответе больше не приходит.

**Не проверено на реальном каталоге.** Структура ответа `catalog.product.list`
предполагается стандартной для REST Б24 (`{ result: { products: [...] },
total, next }`), но некоторые версии/типы каталога могут отдавать
`result` как массив без обёртки `products` — скрипт учитывает оба варианта,
но это стоит перепроверить на первом реальном запуске.
