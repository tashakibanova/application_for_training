# worker/ — Cloudflare Worker для прототипа заявки на обучение

Serverless-прокси между статичной формой (`site/`), Bitrix24 REST и Google
Apps Script (Google Sheets). Схема данных — единый источник правды
`CONTRACT.md` в корне репозитория. Это прототип: минимум зависимостей,
минимум обработки ошибок, чтобы можно было быстро запустить эксперимент.

Эндпоинты:
- `POST /submit` — основная отправка заявки (§2 CONTRACT.md).
- `POST /track` — best-effort отметка о незавершённом заполнении (§3 CONTRACT.md).

---

## 1. Установка зависимостей

```bash
cd worker
npm install
```

`xlsx` (SheetJS Community Edition) в этом `package.json` указан ссылкой на
tarball с `cdn.sheetjs.com`, а не версией из публичного npm-registry.
**Причина:** пакет `xlsx` в npm-registry не обновляется дальше версии
`0.18.5` — актуальные версии SheetJS распространяет только собственный CDN.
Если в вашей сети/CI недоступен `cdn.sheetjs.com` (firewall, политика
безопасности), замените зависимость в `package.json` на:

```json
"xlsx": "^0.18.5"
```

и переустановите (`npm install`) — старая версия из обычного npm тоже
поддерживает нужные нам функции (`utils.aoa_to_sheet`, `utils.book_new`,
`write` с `type: "base64"`), просто без последних фиксов SheetJS.

## 2. Авторизация в Cloudflare

```bash
npx wrangler login
```

Откроет браузер для авторизации в вашем аккаунте Cloudflare (бесплатный
тариф Workers достаточен для объёма теста).

## 3. Секреты (переменные окружения)

Секреты **никогда** не хранятся в `wrangler.toml` и не хардкодятся в коде —
только через `wrangler secret put`. Задайте все три секрета из §5
CONTRACT.md (команда запросит значение интерактивно, ввод не отображается
в истории терминала):

```bash
npx wrangler secret put BITRIX_WEBHOOK_URL
# пример значения: https://ваш-портал.bitrix24.ru/rest/1/xxxxxxxxxxxxxxxx/

npx wrangler secret put SHEETS_WEBHOOK_URL
# URL Apps Script Web App, обычно вида:
# https://script.google.com/macros/s/XXXXX/exec

npx wrangler secret put SHEETS_WEBHOOK_SECRET
# произвольная строка-секрет, которую Apps Script сверяет в поле "secret"
```

`BITRIX_WEBHOOK_URL` — входящий вебхук Б24 с правами `crm` (запись
комментариев и файлов в сделки). Worker сам добавляет к нему
`crm.timeline.comment.add`, поэтому в секрет нужно положить URL **без**
имени метода, оканчивающийся на `/`.

## 4. Настройка CORS для домена формы

Домен(ы) статичной страницы указываются НЕ секретом, а обычной переменной
`ALLOWED_ORIGIN` в `wrangler.toml`:

```toml
[vars]
ALLOWED_ORIGIN = "https://ваш-аккаунт.github.io"
```

Несколько разрешённых origin — через запятую:

```toml
ALLOWED_ORIGIN = "https://ваш-аккаунт.github.io,https://форма.netlify.app"
```

По умолчанию стоит `"*"` (разрешён любой origin) — удобно для первых
тестов, но после того как форма получит постоянный адрес, замените на
конкретный домен(ы), чтобы форму нельзя было встроить на произвольном
чужом сайте. После изменения `wrangler.toml` нужен повторный `wrangler deploy`.

Worker сам обрабатывает preflight `OPTIONS`-запрос и отдаёт
`Access-Control-Allow-Origin` / `-Methods` / `-Headers` на все ответы
`/submit` и `/track`.

## 5. Деплой

```bash
npx wrangler deploy
```

После деплоя wrangler выведет URL воркера вида
`https://zayavka-worker.<ваш-субдомен>.workers.dev`. Этот URL прописывается
в `site/` как адрес для `fetch('.../submit')` и `navigator.sendBeacon('.../track')`.

Локальный запуск для отладки:

```bash
npx wrangler dev
```

Пример проверки вручную (замените `dealId`/поля на осмысленные):

curl -X POST http://127.0.0.1:8787/submit \

## 6. Известные риски и что проверить при реальном деплое

### 6.1 Совместимость библиотеки `xlsx` (SheetJS) с Workers runtime

- Официальная документация SheetJS подтверждает, что модуль работает в
  Cloudflare Workers (`import XLSX from "xlsx"`), но их собственный пример
  использует `XLSX.write(wb, { type: "buffer" })`, что подразумевает наличие
  глобального Node `Buffer`.
- В этом Worker я **сознательно использую `type: "base64"`** вместо
  `"buffer"` — этот режим у SheetJS не зависит от Node `Buffer` (он и в
  браузере, где `Buffer` тоже не существует, работает через собственную
  base64-реализацию библиотеки). Это должно быть безопасно в Workers без
  дополнительных флагов совместимости.
- Как страховка, в `wrangler.toml` всё равно включён флаг
  `compatibility_flags = ["nodejs_compat"]` — он добавляет полифилл `Buffer`
  и других Node-глобалов, если он всё-таки где-то внутри библиотеки
  потребуется.
- **Я не проверял это реальным деплоем** (только по документации SheetJS и
  описанию API `write`). При первом реальном деплое обязательно прогнать
  `POST /submit` с непустым `listeners` и убедиться, что:
  - воркер не падает с ошибкой типа `ReferenceError: Buffer is not defined`
    или `TypeError` внутри `XLSX.write`;
  - файл, приложенный к комментарию в сделке Б24, открывается в Excel/LibreOffice
    без ошибок и содержит корректную кириллицу.
  - Если ошибка всё же возникнет — как запасной вариант используйте
    `XLSX.write(wb, { type: "array", bookType: "xlsx" })` (возвращает
    `Uint8Array`) и конвертируйте в base64 вручную циклом по байтам (без
    `Buffer.from`, который может быть недоступен без `nodejs_compat`).

### 6.2 Формат вложения файла в Bitrix24 REST

- Проверено по официальной документации apidocs.bitrix24.com
  (`crm.timeline.comment.add`): поле `FILES` — это массив пар
  `[имяФайла, base64-содержимое]`, без дополнительной обёртки:
  ```json
  "FILES": [["Заявка.xlsx", "UEsDBBQ..."]]
  ```
  Именно так и реализовано в `src/index.js`. Формулировка `fileContent: [fileName, base64Content]`
  из исходного задания в документации Б24 не встречается — там поле
  называется прямо `FILES`, ключа `fileContent` нет.
- Не проверено реальным вызовом. При первом деплое с `?deal=` в ссылке
  убедитесь, что:
  - комментарий появляется в таймлайне сделки;
  - файл действительно прикреплён и скачивается;
  - `ENTITY_ID` передаётся как число (Worker делает `Number(dealId)` — если
    в вашем портале ID сделки не числовой, потребуется правка).

### 6.3 CPU-лимиты бесплатного тарифа Workers

Генерация xlsx — синхронная CPU-операция. На бесплатном тарифе лимит CPU
времени на вызов ограничен (десятки мс). Для тестового объёма заявок
(единицы-десятки слушателей на заявку) этого более чем достаточно, но если
список слушателей окажется очень большим — стоит проверить, не упирается ли
воркер в лимит (`wrangler tail` покажет ошибку `Exceeded CPU time limit`,
если это произойдёт).

### 6.4 Отказоустойчивость /submit

- Если `dealId` указан, но вызов Б24 не удался (неверный ID сделки,
  недоступен вебхук и т.п.) — Worker **не роняет заявку**, а автоматически
  делает fallback: пишет её в лист «Незакреплённые заявки» с пометкой об
  ошибке Б24, чтобы данные не потерялись (в духе требования брифа "чтобы
  данные не терялись"). Это расширение поведения сверх буквального текста
  CONTRACT.md §2/§4.1 — по формату ответа ничего не меняется
  (`{ "ok": true, "target": "sheet" }` в этом случае).
- Если и это не удалось (Apps Script тоже недоступен) — возвращается
  `{ "ok": false, "error": "bitrix_and_sheet_failed" }` с HTTP 502, чтобы
  фронтенд мог показать пользователю сообщение "заявка не отправлена,
  попробуйте позже/свяжитесь с менеджером".
- Метрики (`sheet: "metrics"`) отправляются через `ctx.waitUntil` —
  асинхронно, не блокируя ответ пользователю. Если этот вызов упадёт,
  ошибка попадёт только в лог воркера (`wrangler tail`), ответ клиенту не
  меняется — по требованию брифа это не критичный путь.

### 6.5 Секреты

`BITRIX_WEBHOOK_URL`, `SHEETS_WEBHOOK_URL`, `SHEETS_WEBHOOK_SECRET` читаются
только из `env.*` и никогда не выводятся в `console.error` (логируются
только текстовые описания ошибок HTTP-статуса/сообщения). Проверьте это при
код-ревью, если будете модифицировать `src/index.js`.
